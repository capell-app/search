# Search — Improvement & Growth Plan

> Package: capell-app/search · Kind: package · Tier: premium · Product group: Capell Search & SEO · Bundle: search-seo · Status: Draft

## 1. Snapshot

`capell-app/search` adds a public frontend search surface (`GET /search`, autocomplete JSON, click-tracking beacon) plus an admin analytics layer (Top/Trending/Zero-result widgets, three overview stats, a settings page). Domain logic lives entirely in `src/Actions` behind a `Capell\Search\Contracts\Search` driver contract with three implementations: `DatabaseSearch` (LIKE/FULLTEXT over a flat table, default `pages`), `SiteDiscoverySearch` (canonical-URL registry metadata), and `ScoutSearch` (multi-source Scout). Key Actions: `RunSearchAction` (orchestrator), `ResolveExpandedSearchQueriesAction` (synonyms + Levenshtein/explicit typo correction), `ApplySearchResultEnhancementsAction` (promotions, source weighting, exact-match/recency/click boosts), `SanitizeSearchResultAction` (URL + meta scrubbing), `RecordSearchAction` (hashed query logging). One table `search_logs` (model `SearchLog`), one settings class `SearchSettings`, one command `search:purge` (scheduled monthly). Runtime deps are light: `lorisleiva/laravel-actions`, `spatie/laravel-data`, `spatie/laravel-package-tools`; Scout/Typesense and `capell-app/site-discovery` are `suggest`-only.

Current marketplace summary: _"Production-grade site search for Capell — relevance-tuned results with synonyms, typo tolerance, and curated promotions, plus admin insights into what visitors search for and where you have no answers."_ Manifest currently declares the extension card plus the settings screen pair. Header-search and insight-widget captures were demoted because they showed settings or blank Insights pages rather than Search UI.

## 2. Improvements (existing functionality)

Prioritized; effort S/M/L.

1. **Done — Fix the outdated `Search` contract example in docs** — `docs/drivers-and-logging.md` "Swap the Search Driver" now imports `SearchFilterData`, shows the full six-argument `Search::search()` signature (`$siteId, $languageId, $filters`), and demonstrates a public-safe escaping `highlight()` implementation. Evidence: `docs/drivers-and-logging.md`, `src/Contracts/Search.php`. — **Shipped**

2. **Multi-query (synonym/typo) pagination over-fetches and mis-counts totals** — `RunSearchAction::searchForQueries()` re-runs the driver once per expanded query at `page:1` with `perPage = perPage * page`, merges, de-dupes by URL, then `min($total, $items->count())`. For deep pages or many synonym variants this multiplies DB/Scout round-trips and produces an approximate `total()` that drifts from the rendered count. Consider a single driver call with OR-combined terms (DB) or `Scout` multi-term, or cap expansion breadth. — `src/Actions/RunSearchAction.php:59` — **M**

3. **`ApplySearchResultEnhancementsAction` recomputes click counts and re-queries on every search** — `clickCounts()` runs a `GROUP BY clicked_result_url` aggregate across the entire `search_logs` table for _every_ frontend search (uncached), and the whole enhancement pipeline maps the collection ~7 times allocating a fresh `SearchResultData` each pass. On a busy site this is the hottest path. Cache `clickCounts()` (short TTL, keyed by site) and collapse the map chain. — `src/Actions/ApplySearchResultEnhancementsAction.php:217` — **M**

4. **Resolve the `Search` driver once, not inside the public Blade** — `resources/views/components/results.blade.php:7` calls `app(Search::class)` from the view to reach `highlight()`. The capell convention is that public Blade receives hydrated render data and does not resolve services. `highlight()` is pure string work; pass pre-highlighted strings (or a tiny highlighter value object) into the view from the controller instead of binding the driver in-template. — `resources/views/components/results.blade.php:7`, `src/Http/Controllers/SearchController.php:59` — **M**

5. **Shipped: `DatabaseSearch::canUseFullText()` memoizes schema detection per connection/table/column set.** The `information_schema.STATISTICS` lookup now runs once per compatible driver/database/table/column combination and caches both positive and fallback results. Evidence: `src/Drivers/DatabaseSearch.php`, `tests/Unit/Search/DatabaseSearchTest.php`. — **S/M**

6. **Shipped: FULLTEXT index matching now accepts covering indexes regardless of order.** Configured search columns only need to be present in one FULLTEXT index, so indexes with additional columns or different ordering no longer silently fall back to LIKE; incomplete indexes still fall back. Evidence: `src/Drivers/DatabaseSearch.php`, `tests/Unit/Search/DatabaseSearchTest.php`. — **M**

7. **Autocomplete reruns the full enhancement pipeline for type-ahead** — `RunAutocompleteSearchAction` calls `RunSearchAction::run()`, which applies promotions, click-boost queries, recency boosts, etc. for a 6-item dropdown on every keystroke (debounced 150ms). A lighter "suggest" path (driver query + minimal mapping, skip click-count aggregate) would cut DB load on the rate-limited public endpoint. — `src/Actions/RunAutocompleteSearchAction.php:53` — **M**

8. **`type_labels` default is hardcoded English** — `config/capell-search.php:54` ships `'page' => 'Page'` and `ApplySearchResultEnhancementsAction::applyTypeLabel()` falls back to `Str::headline($type)`. Neither path is translated, so result-type chips appear in English on non-English sites. Resolve labels via the source registry / translation keys. — `src/Actions/ApplySearchResultEnhancementsAction.php:92` — **S/M**

9. **Click-tracking match is exact-tuple and silently drops on hash/IP change** — `RecordSearchResultClickAction` matches the originating log by `normalized_query + site + language + ip_hash + user_agent_hash` within a 30-min window. If `hash_visitor_data` differs between the search and click request (e.g. proxy IP rotation, or hashing toggled), no log matches and the click is lost with no signal. Consider a server-issued opaque search token echoed by the beacon instead of re-deriving the tuple. — `src/Actions/RecordSearchResultClickAction.php:42` — **M**

10. **`SearchSettings` exposes no UI for the rich config knobs** — Synonyms, typo corrections/terms, promoted results, source weights, ranking boosts, and database column mapping are config-file-only (`config/capell-search.php`); the Filament settings schema (`SearchSettingsSchema`) only covers enabled/driver/per-page/logging/retention/hashing/min-length/source toggles. A premium "search insights" buyer expects to edit synonyms and promoted results in the admin, not redeploy. — `src/Filament/Settings/SearchSettingsSchema.php` — **L**

## 3. Missing Features (gaps)

Tied to `capabilities[]` (`search`, `search-admin/console/frontend`, `search-synonyms`, `search-promoted-results`, `search-typo-corrections`, `search-source-weighting`, `search-zero-result-reporting`, `search-site-discovery-indexing`) and search-category norms.

- **Faceted filtering UI (table-stakes).** `SearchFilterData` (types + sourceKeys) and `NormalizeSearchFiltersAction` exist and are honored by every driver, but no frontend renders facet checkboxes/counts and the results page (`pages/search.blade.php`) shows none. The plumbing is built; the surface is missing. **Differentiator-adjacent** once it shows live result counts per facet.
- **No admin management of synonyms / promoted results / typo dictionaries.** Advertised capabilities (`search-synonyms`, `search-promoted-results`, `search-typo-corrections`) are config-only. For a premium tier these belong in the admin as editable lists (see §2.10). **Differentiator** — most CMS search add-ons can't curate results.
- **No saved/persisted search analytics beyond raw logs.** Zero-result reporting exists as a widget, but there is no "create a synonym/promotion from this zero-result term" action closing the loop from insight → fix. **High-value differentiator.**
- **Autocomplete suggestions are result-echo only.** `RunAutocompleteSearchAction` returns trimmed search results, not query _suggestions_ (popular completions, did-you-mean). Real type-ahead suggests _queries_ from `search_logs` (top normalized queries by prefix). **Table-stakes for "autocomplete".**
- **Did-you-mean / spelling suggestion surface.** Typo correction silently rewrites the query server-side, but the user is never told ("Showing results for _capell_ — search instead for _capel_"). Both correction and the surfaced suggestion are search norms. **Table-stakes.**
- **Multi-driver Scout breadth is shallow.** `ScoutSearch` paginates each source independently then merges/sorts in PHP by naive `substr_count` score (`mapModelToResult` → `score()`), discarding the engine's relevance ranking and `total`. For Meilisearch/Typesense this throws away the engine's strength. Honor engine scores/`total` and push filtering down. **Differentiator** for the premium Scout story. — `src/Drivers/ScoutSearch.php:147`
- **No indexing-freshness or queue story for Scout.** `config('capell-search.scout.chunk')` is defined but unused; there is no `search:index`/`search:flush` command, no observer wiring docs, and no staleness signal. The manifest advertises `search-site-discovery-indexing` (registry coverage) but not content-index freshness. **Table-stakes** for Scout buyers.
- **Relevance tuning is fixed-formula, not weighted/learnable.** Boosts (exact 25.0, recency, click 0.25) are global config scalars; there is no per-field weighting (title vs body) in `DatabaseSearch` and no per-source relevance config beyond a single `weight`. **Differentiator** if exposed as tunable field weights.
- **No runtime health probe.** The four `healthChecks` in `capell.json` point at `SearchHealthCheck`, which is a compile-time marker (only `compatibleCapellApiVersion()`); it performs **no runtime verification** (see §4). A real probe (driver resolves, index reachable, recent log writes succeeding) is a gap for a premium "priority support" SKU.

## 4. Issues / Risks

- **Stub-by-convention health check that cannot detect a broken search.** `src/Health/SearchHealthCheck.php` implements `ChecksExtensionHealth` but only returns `'^4.0'`. The four detailed `capell.json` health labels ("Frontend search normalizes queries and returns paginated results", "Database search escapes wildcards and clamps pagination arguments", "Search logs capture…", "purge enforces retention") describe what the **test suite** asserts, not anything executed at runtime. This is the project's marker pattern (mirrors `SearchFrontendRouteContribution`), so it is not a lie — but it means operators get no runtime detection if Scout/Meili is down or the `pages` driver points at the wrong columns. — `src/Health/SearchHealthCheck.php`
- **Default driver targets a table that cannot meaningfully search page content.** `config/capell-search.php:93` defaults `database.table => 'pages'` with `columns => ['title','excerpt','body']`. Capell pages store body content in nested/translated structures, not flat `body`/`excerpt` columns; the package's own docs and `capell.json` "Common Pitfalls" admit this. Out-of-the-box the default driver returns near-empty results on a real Capell install — a correctness/UX risk masked by tests that seed a synthetic flat `pages` table (`tests/Unit/Search/DatabaseSearchTest.php:18`). — `src/Drivers/DatabaseSearch.php:37`
- **Result-visibility safety depends entirely on upstream, with no enforcement in `ScoutSearch`.** `DatabaseSearch` filters `status = published` and applies site/language scoping _only when those columns exist_ (`applyContextFilters`, silently skipped otherwise). `ScoutSearch` applies **no** published/visibility predicate at all — it trusts each model's `toSearchableArray()`/index to exclude drafts and private content (`searchSource`, `src/Drivers/ScoutSearch.php:114`). If a host indexes a model without excluding unpublished rows, drafts leak into public results. There is no test proving unpublished/private exclusion for the Scout path, and no contract-level guard. **Multi-site leakage:** Scout site scoping is a best-effort `where('site_id', …)` only if the builder supports it. — `src/Drivers/ScoutSearch.php:114`
- **Public-output scrubbing is allow-list-by-substring and order-dependent.** `SanitizeSearchResultAction::safeMeta()` rejects meta keys containing `admin/editor/model/signed/token/preview/field`; a key like `internal_ref` or `owner_id` passes through into `SearchResultData->meta` (though current Blade never renders `meta`). `sanitizeUrl()` blocks `/admin`, `/cp`, `/filament` prefixes from config — reasonable, but the block list is config-overridable and applied after scheme parse. Net: safe today because views don't print `meta`, but the meta channel is an unguarded leakage vector if a future view renders it. — `src/Actions/SanitizeSearchResultAction.php:144`
- **Shipped guardrail — `highlight()` must escape before adding markup.** `results.blade.php:53-57` prints `{!! $search->highlight(...) !!}`. All three built-in drivers `htmlspecialchars`/`e()` the text before wrapping `<mark>`, tests assert escaping, and `src/Contracts/Search.php` now documents that custom implementations must escape the full input text and only add trusted `<mark>` markup. Evidence: `src/Contracts/Search.php`, `docs/drivers-and-logging.md`, `tests/Unit/Search/DatabaseSearchTest.php`, `tests/Unit/Search/ScoutSearchTest.php`, `tests/Unit/Search/SiteDiscoverySearchTest.php`.
- **Shipped: click-tracking beacon no longer depends on cached-page CSRF state.** `search/click` is now explicitly CSRF-exempt and throttled through `capell-search.click_tracking.rate_limiter`; the header and result click beacons post without a meta CSRF token, and the result component owns a guarded beacon listener so search-result pages still track clicks when the header search modal is absent. Evidence: `routes/web.php`, `config/capell-search.php`, `resources/views/components/header/search-dialog.blade.php`, `resources/views/components/results.blade.php`, `tests/Feature/Http/SearchControllerTest.php`.
- **Shipped guardrail — search log writes are deferred after response.** `capell.json` sets `frontendRenderBudgetMs: 20` and `adminQueryBudget: 40`. The frontend path still does driver query (×N expansions) plus a full `search_logs` click-count aggregate, but the analytics insert is no longer inline: `SearchController` now calls `RecordSearchAction::dispatchAfterResponse()` with scalar visitor metadata, and `RecordSearchAction` still owns the actual write. Evidence: `src/Http/Controllers/SearchController.php`, `src/Actions/RecordSearchAction.php`, `tests/Feature/Http/SearchControllerTest.php`, `tests/Feature/Actions/RecordSearchActionTest.php`. Cache the aggregate next (§2.3). `cacheSafety.cacheable=false` is correct (per-query).
- **i18n gaps.** `type_labels` default and the `Str::headline` fallback are untranslated (§2.8); `SiteDiscoverySearch::titleFromUrl()` returns a literal `'Home'` (`src/Drivers/SiteDiscoverySearch.php:135`) and derives titles from URL slugs via `Str::headline`, producing English-ish titles regardless of locale.
- **Test coverage — covered:** `DatabaseSearch` (escape, clamp, site/lang/status filter, pagination, highlight, FULLTEXT index compatibility), `ScoutSearch` (registry merge, weight, absolute URL, highlight) via in-memory fakes, `SiteDiscoverySearch` (indexability, metadata match, paginate/highlight), `RecordSearchAction`/`RecordSearchResultClickAction`, `PurgeSearchLogsAction`, `ResolveExpandedSearchQueries` (synonyms + typos), `ApplySearchResultEnhancements` (promotions + weights + dedupe), `RunSearchAction` (synonym dedupe), `SearchController` (autocomplete limits, blank query, normalized passthrough, "no package identifiers in markup"), click-beacon route CSRF exemption and tokenless header/results markup, provider binding/driver selection, settings defaults, manifest requirements, insights actions, widgets. **Not covered:** the full-page render path (`canRenderFrontendShell()` → `capell::app` shell, `src/Http/Controllers/SearchController.php:61-77` — only the bare-content and configured-page-view branches are tested); the `click` POST endpoint end-to-end; MySQL execution of the FULLTEXT branch (tests use SQLite and unit-test compatibility logic directly); rate-limiter enforcement under load; `SearchGeneratedOutputCoverageSource`.
- **Minor manifest/dep mismatches.** `composer.json` requires `php: ^8.3` while the platform standard is 8.4; `BuildTopClickedResultsQueryAction` is wired into `SearchOverviewStatsWidget` and tested but is **not** listed in `capell.json` `actions[]` (the other `Build*` query actions aren't either, so the map is selectively incomplete). `drivers-and-logging.md` config table omits the `driver` value `site_discovery` (says "`database` or `scout`") despite it being a first-class, tested driver.

## 5. Marketplace & Selling

**Shipped copy refresh.** `composer.json`, `capell.json`, and `resources/lang/en/package.php` now lead with the production search value proposition instead of the old feature-dump copy. Evidence: composer description _"Production-grade site search for Capell with relevance-tuned results, synonyms, typo tolerance, curated promotions, and admin search insights."_ Marketplace summary _"Production-grade site search for Capell — relevance-tuned results with synonyms, typo tolerance, and curated promotions, plus admin insights into what visitors search for and where you have no answers."_

**Improved one-sentence summary:**

> Production-grade site search for Capell — relevance-tuned results with synonyms, typo tolerance, and curated promotions, plus admin insights into what visitors search for and where you have no answers.

**Improved 3–4 sentence description:**

> Search gives every Capell site a fast, themeable search experience: a results page, header search field, and type-ahead autocomplete, backed by a pluggable driver contract (database, Site Discovery URL registry, or Laravel Scout/Meilisearch/Typesense). Curate results with synonyms, typo corrections, and promoted "best bet" answers, and tune ranking with source weighting plus exact-match, recency, and click-through boosts. The admin dashboard surfaces top searches, trending terms, and — critically — zero-result queries, so editors see exactly where content is missing. Privacy-first by default: visitor identifiers are hashed and query logs auto-purge on a configurable retention window.

**Reopened screenshot/media promotion.** `capell.json.marketplace.screenshots` now lists the extension card plus the settings screen pair. Recapture frontend search, header search, top-searches, trending-searches, and zero-result widget screenshots before promotion. Add one annotated "curation" screenshot (synonyms/promotions) once §2.10 lands, since result curation is the headline differentiator.

**Pricing / tier / bundle positioning.** `premium` tier in the `search-seo` bundle is right: search + SEO is a natural "growth" pairing. Position Search as the **demand-side** half (what visitors look for) against `seo-suite` as the **supply-side** half (how content is found by engines). Cross-sell is already wired: `suggest` lists `capell-app/site-discovery` (URL-registry indexing + generated-output coverage) and `docs/README.md` "Read Next" points at `seo-suite` and `site-discovery`. Make the bundle explicit: **Search + SEO Suite + Site Discovery = "Findability"**. Zero-result analytics is the upsell hook into a future Knowledge-Base / content-gap workflow (a zero-result term → "create article" CTA bridges Search and `knowledge-base`).

**Differentiators / value props / target buyer.** Differentiators vs. table-stakes CMS search: (1) editable result _curation_ (synonyms, promotions, typo dictionaries) — most CMS search can't do this; (2) zero-result + trending analytics that turn search into a content-strategy signal; (3) driver-agnostic contract so a site starts on DB search and graduates to Typesense without touching the frontend. Target buyer: agencies and content teams running multi-site Capell installs who treat on-site search as a conversion/retention surface, not a checkbox.
