## 1. Snapshot

`capell-app/search` adds a public frontend search surface (`GET /search`, autocomplete JSON, click-tracking beacon) plus an admin analytics layer (Top/Trending/Zero-result widgets, three overview stats, a settings page). Domain logic lives entirely in `src/Actions` behind a `Capell\Search\Contracts\Search` driver contract with three implementations: `DatabaseSearch` (LIKE/FULLTEXT over a flat table, default `pages`), `SiteDiscoverySearch` (canonical-URL registry metadata), and `ScoutSearch` (multi-source Scout). Key Actions: `RunSearchAction` (orchestrator), `ResolveExpandedSearchQueriesAction` (synonyms + Levenshtein/explicit typo correction), `ApplySearchResultEnhancementsAction` (promotions, source weighting, exact-match/recency/click boosts), `SanitizeSearchResultAction` (URL + meta scrubbing), `RecordSearchAction` (hashed query logging). One table `search_logs` (model `SearchLog`), one settings class `SearchSettings`, one command `search:purge` (scheduled monthly). Runtime deps are light: `lorisleiva/laravel-actions`, `spatie/laravel-data`, `spatie/laravel-package-tools`; Scout/Typesense and `capell-app/site-discovery` are `suggest`-only.

Current marketplace summary (verbatim from `capell.json`): _"Search adds a frontend search route, configurable search drivers, synonym expansion, typo correction, promoted results, source weighting, result click tracking, zero-result reporting, query logging, Site Discovery indexing, and admin search insights."_ Manifest declares **1** marketplace screenshot (`docs/assets/marketplace/extension-card.jpg`), but `docs/screenshots/` ships **12** images (6 use-cases Ă— light/dark) and `docs/screenshots.json` defines 6 entries â€” a manifest/media mismatch (the marketplace media array is under-populated).

## 2. Improvements (existing functionality)

Prioritized; effort S/M/L.

1. **Fix the outdated `Search` contract example in docs** â€” `docs/drivers-and-logging.md` "Swap the Search Driver" shows `search(string $query, int $perPage = 10, int $page = 1)`, but the real contract (`src/Contracts/Search.php`) has six parameters (`$siteId, $languageId, $filters`). A copy-pasted custom driver will fatal with an abstract-method/signature mismatch. â€” `docs/drivers-and-logging.md` â€” **S**

2. **Multi-query (synonym/typo) pagination over-fetches and mis-counts totals** â€” `RunSearchAction::searchForQueries()` re-runs the driver once per expanded query at `page:1` with `perPage = perPage * page`, merges, de-dupes by URL, then `min($total, $items->count())`. For deep pages or many synonym variants this multiplies DB/Scout round-trips and produces an approximate `total()` that drifts from the rendered count. Consider a single driver call with OR-combined terms (DB) or `Scout` multi-term, or cap expansion breadth. â€” `src/Actions/RunSearchAction.php:59` â€” **M**

3. **`ApplySearchResultEnhancementsAction` recomputes click counts and re-queries on every search** â€” `clickCounts()` runs a `GROUP BY clicked_result_url` aggregate across the entire `search_logs` table for _every_ frontend search (uncached), and the whole enhancement pipeline maps the collection ~7 times allocating a fresh `SearchResultData` each pass. On a busy site this is the hottest path. Cache `clickCounts()` (short TTL, keyed by site) and collapse the map chain. â€” `src/Actions/ApplySearchResultEnhancementsAction.php:217` â€” **M**

4. **Resolve the `Search` driver once, not inside the public Blade** â€” `resources/views/components/results.blade.php:7` calls `app(Search::class)` from the view to reach `highlight()`. The capell convention is that public Blade receives hydrated render data and does not resolve services. `highlight()` is pure string work; pass pre-highlighted strings (or a tiny highlighter value object) into the view from the controller instead of binding the driver in-template. â€” `resources/views/components/results.blade.php:7`, `src/Http/Controllers/SearchController.php:59` â€” **M**

5. **`DatabaseSearch::canUseFullText()` runs an `information_schema.STATISTICS` query on every search** â€” Detecting the FULLTEXT index by querying `information_schema` per request is avoidable overhead and requires the connection user to have schema-stats access. Memoize per-connection/table, or make FULLTEXT opt-in via config (`capell-search.database.fulltext => true`) since the index migration already exists. â€” `src/Drivers/DatabaseSearch.php:197` â€” **S/M**

6. **FULLTEXT exact-column-set match is brittle** â€” `canUseFullText()` only returns true when the configured `$columns` array is byte-identical to an index's column set (`$indexColumns === $columns`). Any extra/missing column, or order difference, silently falls back to LIKE, defeating the shipped FULLTEXT migration without warning. Match on subset/superset or log when fallback occurs. â€” `src/Drivers/DatabaseSearch.php:219` â€” **M**

7. **Autocomplete reruns the full enhancement pipeline for type-ahead** â€” `RunAutocompleteSearchAction` calls `RunSearchAction::run()`, which applies promotions, click-boost queries, recency boosts, etc. for a 6-item dropdown on every keystroke (debounced 150ms). A lighter "suggest" path (driver query + minimal mapping, skip click-count aggregate) would cut DB load on the rate-limited public endpoint. â€” `src/Actions/RunAutocompleteSearchAction.php:53` â€” **M**

8. **`type_labels` default is hardcoded English** â€” `config/capell-search.php:54` ships `'page' => 'Page'` and `ApplySearchResultEnhancementsAction::applyTypeLabel()` falls back to `Str::headline($type)`. Neither path is translated, so result-type chips appear in English on non-English sites. Resolve labels via the source registry / translation keys. â€” `src/Actions/ApplySearchResultEnhancementsAction.php:92` â€” **S/M**

9. **Click-tracking match is exact-tuple and silently drops on hash/IP change** â€” `RecordSearchResultClickAction` matches the originating log by `normalized_query + site + language + ip_hash + user_agent_hash` within a 30-min window. If `hash_visitor_data` differs between the search and click request (e.g. proxy IP rotation, or hashing toggled), no log matches and the click is lost with no signal. Consider a server-issued opaque search token echoed by the beacon instead of re-deriving the tuple. â€” `src/Actions/RecordSearchResultClickAction.php:42` â€” **M**

10. **`SearchSettings` exposes no UI for the rich config knobs** â€” Synonyms, typo corrections/terms, promoted results, source weights, ranking boosts, and database column mapping are config-file-only (`config/capell-search.php`); the Filament settings schema (`SearchSettingsSchema`) only covers enabled/driver/per-page/logging/retention/hashing/min-length/source toggles. ## 3. Missing Features (gaps)

Tied to `capabilities[]` (`search`, `search-admin/console/frontend`, `search-synonyms`, `search-promoted-results`, `search-typo-corrections`, `search-source-weighting`, `search-zero-result-reporting`, `search-site-discovery-indexing`) and search-category norms.

- **Faceted filtering UI (table-stakes).** `SearchFilterData` (types + sourceKeys) and `NormalizeSearchFiltersAction` exist and are honored by every driver, but no frontend renders facet checkboxes/counts and the results page (`pages/search.blade.php`) shows none. The plumbing is built; the surface is missing. **Differentiator-adjacent** once it shows live result counts per facet.
- **No admin management of synonyms / promoted results / typo dictionaries.** Advertised capabilities (`search-synonyms`, `search-promoted-results`, `search-typo-corrections`) are config-only. For a premium tier these belong in the admin as editable lists (see Â§2.10). **Differentiator** â€” most CMS search add-ons can't curate results.
- **No saved/persisted search analytics beyond raw logs.** Zero-result reporting exists as a widget, but there is no "create a synonym/promotion from this zero-result term" action closing the loop from insight â†’ fix. **High-value differentiator.**
- **Autocomplete suggestions are result-echo only.** `RunAutocompleteSearchAction` returns trimmed search results, not query _suggestions_ (popular completions, did-you-mean). Real type-ahead suggests _queries_ from `search_logs` (top normalized queries by prefix). **Table-stakes for "autocomplete".**
- **Did-you-mean / spelling suggestion surface.** Typo correction silently rewrites the query server-side, but the user is never told ("Showing results for _capell_ â€” search instead for _capel_"). Both correction and the surfaced suggestion are search norms. **Table-stakes.**
- **Multi-driver Scout breadth is shallow.** `ScoutSearch` paginates each source independently then merges/sorts in PHP by naive `substr_count` score (`mapModelToResult` â†’ `score()`), discarding the engine's relevance ranking and `total`. For Meilisearch/Typesense this throws away the engine's strength. Honor engine scores/`total` and push filtering down. **Differentiator** for the premium Scout story. â€” `src/Drivers/ScoutSearch.php:147`
- **No indexing-freshness or queue story for Scout.** `config('capell-search.scout.chunk')` is defined but unused; there is no `search:index`/`search:flush` command, no observer wiring docs, and no staleness signal. The manifest advertises `search-site-discovery-indexing` (registry coverage) but not content-index freshness. **Table-stakes** for Scout buyers.
- **Relevance tuning is fixed-formula, not weighted/learnable.** Boosts (exact 25.0, recency, click 0.25) are global config scalars; there is no per-field weighting (title vs body) in `DatabaseSearch` and no per-source relevance config beyond a single `weight`. **Differentiator** if exposed as tunable field weights.
- **No runtime health probe.** The four `healthChecks` in `capell.json` point at `SearchHealthCheck`, which is a compile-time marker (only `compatibleCapellApiVersion()`); it performs **no runtime verification** (see Â§4). A real probe (driver resolves, index reachable, recent log writes succeeding) is a gap for a premium "priority support" SKU.

## 4. Issues / Risks

- **Stub-by-convention health check that cannot detect a broken search.** `src/Health/SearchHealthCheck.php` implements `ChecksExtensionHealth` but only returns `'^4.0'`. The four detailed `capell.json` health labels ("Frontend search normalizes queries and returns paginated results", "Database search escapes wildcards and clamps pagination arguments", "Search logs captureâ€¦", "purge enforces retention") describe what the **test suite** asserts, not anything executed at runtime. This is the project's marker pattern (mirrors `SearchFrontendRouteContribution`), so it is not a lie â€” but it means operators get no runtime detection if Scout/Meili is down or the `pages` driver points at the wrong columns. â€” `src/Health/SearchHealthCheck.php`
- **Default driver targets a table that cannot meaningfully search page content.** `config/capell-search.php:93` defaults `database.table => 'pages'` with `columns => ['title','excerpt','body']`. Capell pages store body content in nested/translated structures, not flat `body`/`excerpt` columns; the package's own docs and `capell.json` "Common Pitfalls" admit this. Out-of-the-box the default driver returns near-empty results on a real Capell install â€” a correctness/UX risk masked by tests that seed a synthetic flat `pages` table (`tests/Unit/Search/DatabaseSearchTest.php:18`). â€” `src/Drivers/DatabaseSearch.php:37`
- **Result-visibility safety depends entirely on upstream, with no enforcement in `ScoutSearch`.** `DatabaseSearch` filters `status = published` and applies site/language scoping _only when those columns exist_ (`applyContextFilters`, silently skipped otherwise). `ScoutSearch` applies **no** published/visibility predicate at all â€” it trusts each model's `toSearchableArray()`/index to exclude drafts and private content (`searchSource`, `src/Drivers/ScoutSearch.php:114`). If a host indexes a model without excluding unpublished rows, drafts leak into public results. There is no test proving unpublished/private exclusion for the Scout path, and no contract-level guard. **Multi-site leakage:** Scout site scoping is a best-effort `where('site_id', â€¦)` only if the builder supports it. â€” `src/Drivers/ScoutSearch.php:114`
- **Public-output scrubbing is allow-list-by-substring and order-dependent.** `SanitizeSearchResultAction::safeMeta()` rejects meta keys containing `admin/editor/model/signed/token/preview/field`; a key like `internal_ref` or `owner_id` passes through into `SearchResultData->meta` (though current Blade never renders `meta`). `sanitizeUrl()` blocks `/admin`, `/cp`, `/filament` prefixes from config â€” reasonable, but the block list is config-overridable and applied after scheme parse. Net: safe today because views don't print `meta`, but the meta channel is an unguarded leakage vector if a future view renders it. â€” `src/Actions/SanitizeSearchResultAction.php:144`
- **`highlight()` emits unescaped `{!! !!}` â€” XSS-safe only because each driver escapes first.** `results.blade.php:53-57` prints `{!! $search->highlight(...) !!}`. All three drivers `htmlspecialchars`/`e()` the text before wrapping `<mark>`, and tests assert escaping, so it is safe â€” but the safety is distributed across three driver implementations with no shared contract guarantee. A fourth custom driver (the documented extension point) could forget to escape and introduce stored XSS. Document the escaping requirement on the contract and/or escape centrally. â€” `src/Contracts/Search.php:29`, `resources/views/components/results.blade.php:53`
- **Click-tracking beacon is a CSRF-protected POST under the `web` group.** `routes/web.php:18` registers `search/click` as `POST` with `web` middleware (CSRF) and no exemption. The frontend beacon must send a valid CSRF token; on fully cached/static HTML pages (Capell's `frontend.cache` serves static files) the token may be stale/absent, so click logging can silently fail in exactly the high-traffic scenario it's meant to measure. Verify token delivery on cached pages or use a signed tokenless endpoint. â€” `src/Http/Controllers/SearchController.php:87`, `routes/web.php:18`
- **Performance budgets are aggressive and unverified at runtime.** `capell.json` sets `frontendRenderBudgetMs: 20` and `adminQueryBudget: 40`. The frontend path does: driver query (Ă—N expansions) + a full `search_logs` click-count aggregate + a synchronous `RecordSearchAction` insert (`SearchController` calls `RecordSearchAction::run()` inline, not queued â€” `src/Http/Controllers/SearchController.php:48`). 20ms is unlikely under the multi-query + click-aggregate + write path on real data. Move log writes to a queued/`afterResponse` dispatch and cache the aggregate (Â§2.3). `cacheSafety.cacheable=false` is correct (per-query), but the inline write is the budget risk.
- **i18n gaps.** `type_labels` default and the `Str::headline` fallback are untranslated (Â§2.8); `SiteDiscoverySearch::titleFromUrl()` returns a literal `'Home'` (`src/Drivers/SiteDiscoverySearch.php:135`) and derives titles from URL slugs via `Str::headline`, producing English-ish titles regardless of locale.
- **Test coverage â€” covered:** `DatabaseSearch` (escape, clamp, site/lang/status filter, pagination, highlight), `ScoutSearch` (registry merge, weight, absolute URL, highlight) via in-memory fakes, `SiteDiscoverySearch` (indexability, metadata match, paginate/highlight), `RecordSearchAction`/`RecordSearchResultClickAction`, `PurgeSearchLogsAction`, `ResolveExpandedSearchQueries` (synonyms + typos), `ApplySearchResultEnhancements` (promotions + weights + dedupe), `RunSearchAction` (synonym dedupe), `SearchController` (autocomplete limits, blank query, normalized passthrough, "no package identifiers in markup"), provider binding/driver selection, settings defaults, manifest requirements, insights actions, widgets. **Not covered:** the full-page render path (`canRenderFrontendShell()` â†’ `capell::app` shell, `src/Http/Controllers/SearchController.php:61-77` â€” only the bare-content and configured-page-view branches are tested); the `click` POST endpoint end-to-end incl.
